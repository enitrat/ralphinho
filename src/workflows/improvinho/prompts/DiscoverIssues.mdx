## Shared Discovery Standard

- High signal only. Report objective issues with concrete evidence, not style preferences or vague suggestions.
- Read the in-scope files before concluding. Do not infer architecture from one snippet if the surrounding module answers the question.
- Ask: how does this codebase already solve similar problems? Reuse existing patterns when they are healthy.
- Search for existing utilities, helpers, selectors, mappers, guards, and shared logic before recommending a new helper or abstraction.
- Treat unnecessary complexity as a first-class defect.
- Prefer deleting code, collapsing layers, or reusing existing utilities over introducing new abstractions.
- Trust the type system and validated boundaries; do not reward code for defending against impossible states.
- Keep framework-specific assumptions out of your reasoning unless the code in scope clearly requires them.

## Shared Review Checklist

- Reuse and duplication:
  - Flag duplicated helpers, repeated inline transformations, and local inventions that should use an existing shared pattern.
- Simplicity and design:
  - Flag unnecessary indirection, over-engineering, dead options, dead files, and abstractions with only one trivial consumer.
- Correctness and efficiency:
  - Flag unnecessary work, repeated reads/computation, missed concurrency, hot-path bloat, and TOCTOU-style pre-checks that should operate directly and handle failure.
- Tests:
  - If tests are in scope, flag tests that only prove types, exports, naming, or file organization.
  - Prefer tests that validate observable behavior and real public contracts.
- Architecture:
  - Flag leaky abstractions, poor ownership boundaries, and logic that lives in the wrong layer.
