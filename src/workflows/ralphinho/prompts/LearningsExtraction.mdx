## What to Extract

Focus on patterns that are **generalizable across units**, not unit-specific details.

**Good learnings:**
- "Tests that only verify type exports are waste — the compiler handles this" (category: testing)
- "Functions in src/utils/ already handle path normalization — check before writing your own" (category: code-quality)
- "Error paths in API handlers consistently miss logging" (category: error-handling)

**Bad learnings (too specific, skip these):**
- "Fixed a typo in line 42 of foo.ts"
- "Added missing import for Bar"

## Steps

1. Read the review feedback above
2. Identify patterns that a future implementer should know BEFORE writing code
3. For each pattern, classify its category and whether it's likely one-off or recurring
4. Write the learnings file to {props.learningsFilePath}
5. Return the structured output

## File Format

Write {props.learningsFilePath} as markdown:

```markdown
# Learnings: {props.unitName}

## Patterns

### [category] Pattern title
Description of the pattern and why it matters.
Example: concrete code or file reference if applicable.
Frequency: one-off | recurring
```
